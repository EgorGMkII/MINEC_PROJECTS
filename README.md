# CCT RAG Self-Check Service

Сервис для внутренней самопроверки проектов изменений в государственные программы. Это не официальный ответ согласующего органа, а рабочий инструмент, который помогает заранее найти замечания по проекту, приложениям и сопроводительным материалам.

## Что делает сервис

Пользователь загружает:

- пакет проекта изменений: основной проект и приложения;
- региональную программу: базовый документ;
- федеральную программу;
- дополнительные документы: пояснительная записка, ФЭО, отдельные порядки и другие материалы.

Отдельно на сервере лежит скрытый корпус:

- `npa/` — нормативные акты для retrieval;
- `УФАС/UFAS.txt` — правила для отдельной быстрой проверки типовых ошибок.

Эти папки не должны храниться в публичном репозитории. Для локального запуска и сборки Docker-образа их нужно положить рядом с кодом вручную.

Текущий веб-вход находится в `rag_web.py`, основной pipeline — в `summary_rag.py`.

## Архитектура

Основные файлы:

- `rag_web.py` — FastAPI-приложение и HTML-интерфейс.
- `rag_web_public.py` — public-версия приложения.
- `summary_rag.py` — основной pipeline обработки.
- `simple_rag.py` — общие функции парсинга, чанкинга и BM25.
- `rag_support.py` — основной prompt, парсинг файлов, служебные helper-функции.
- `old_prompt.py` — public prompt для более официального проекта заключения.
- `summary_prompts.py` — prompt’ы для суммаризации.
- `ufas_prompts.py` — prompt для UFAS precheck.

## Пайплайн

### 1. Загрузка и парсинг документов

Все загруженные файлы приводятся к plain text.

Поддерживаются:

- `docx`
- `pdf`
- `txt`
- `xlsx`

Для `xlsx` листы читаются построчно и превращаются в текстовые блоки, чтобы дальше их можно было использовать в retrieval и prompt’ах.

### 2. Сбор project-пакета

Все документы из слота проекта изменений:

- сохраняются поштучно как отдельные `project_sources`;
- одновременно склеиваются в один общий `project_text`.

Это нужно потому, что разные этапы pipeline используют разные представления:

- `project_text` нужен для UFAS precheck;
- поштучные `project_sources` нужны для пофайловой суммаризации.

### 3. UFAS precheck

Перед основным RAG запускается отдельная короткая проверка:

- берётся полный `project_text`;
- к нему добавляются правила из `УФАС/UFAS.txt`;
- LLM ищет типовые ошибки формулировок, критериев, ограничений и правоприменения.

Результат:

- короткий блок предварительных замечаний;
- он потом приклеивается к итоговому ответу;
- он кэшируется, чтобы не тратить токены повторно на тот же пакет.

### 4. Суммаризация project-документов

Каждый файл из слота проекта изменений суммаризируется отдельно.

Задача этой суммаризации:

- сохранить ключевые изменения;
- выделить изменения в целях, задачах, структуре программы, показателях и результатах;
- не потерять смысл приложений;
- не утонуть в длинном нормативном тексте.

### 5. Суммаризация всего project-пакета

После пофайловых summaries создаётся единый `project_package_summary`.

Это сводка всего пакета изменений:

- основной проект;
- приложения;
- проекты приложений;
- иные документы из project-слота.

Именно этот summary используется как основа для retrieval query.

### 6. Суммаризация дополнительных документов

Документы из слота `Дополнительные документы` тоже суммаризируются по одному.

Сейчас их summaries:

- участвуют в retrieval;
- также могут попасть в финальный prompt.

Сырые `extra`-документы в retrieval не используются, чтобы не раздувать корпус лишним шумом.

### 7. Формирование retrieval-корпуса

В retrieval-корпус идут:

- `npa` — обычными чанками;
- `base_program` — обычными чанками;
- `federal` — обычными чанками;
- `extra_summary` — summaries дополнительных документов.

Что не идёт в retrieval:

- полный `project_text`;
- сырые `project`-документы;
- сырые `extra`-документы.

### 8. Чанкование

Обычные документы режутся стандартным чанкером.

Summaries режутся отдельно и более крупно, чтобы лучше сохранять связный смысл.

Это уменьшает риск того, что короткий summary распадётся на слишком мелкие и бесполезные куски.

### 9. Retrieval

Запрос к retriever строится по `project_package_summary`, а не по полному тексту проекта.

Используется BM25. После этого включается дополнительный этап покрытия источников:

- сначала выбираются лучшие чанки по общему score;
- действует ограничение на количество чанков от одного документа;
- затем для документов, которые вообще не попали в выборку, добирается хотя бы один лучший чанк.

Итоговая цель:

- не дать одному документу занять всё окно контекста;
- обеспечить хотя бы минимальное покрытие по всем источникам.

### 10. Финальный prompt

В финальный prompt идут:

- основной system prompt самопроверки;
- `project_package_summary`;
- summaries отдельных project-документов;
- summaries дополнительных документов;
- retrieved context.

В prompt не идёт полный текст project-пакета целиком. Вместо этого используется сжатое представление, чтобы модель меньше тонула в шуме длинных нормативных формулировок.

### 11. Финальная генерация ответа

LLM формирует итоговую самопроверку в рабочем стиле:

- тезисно;
- без канцелярской стилистики официального заключения;
- с указанием ошибки;
- с пояснением, почему это важно;
- с рекомендацией, что проверить и как исправить.

К итоговому ответу приклеивается UFAS precheck.

## Кэширование

Кэшируются:

- summaries project-документов;
- `project_package_summary`;
- summaries дополнительных документов;
- результат UFAS precheck.

Это нужно, чтобы не тратить токены повторно на одни и те же документы.

## Локальный запуск

Установить зависимости:

```powershell
python -m pip install -r requirements.txt
```

Создать локальный env-файл, например `.env`:

```text
OPENAI_API_KEY=your_real_key
OPENAI_BASE_URL=https://api.proxyapi.ru/openai/v1
OPENAI_MODEL=gpt-5.3-chat-latest
```

`.env` не должен попадать в git. Без `OPENAI_API_KEY` pipeline не сможет вызвать LLM.

Перед запуском положить рядом с кодом скрытые папки:

```text
npa/
УФАС/
```

Ожидаемая структура:

```text
npa/
  *.docx
УФАС/
  UFAS.txt
```

В репозитории хранятся только пустые папки-заглушки. Содержимое этих папок локальное и игнорируется git.

Запустить developer-версию:

```powershell
python -m uvicorn rag_web:app --reload --port 8010
```

Запустить public-версию:

```powershell
python -m uvicorn rag_web_public:app --reload --port 8011
```

Открыть:

```text
http://127.0.0.1:8010/
http://127.0.0.1:8011/
```

## Docker

Сборка:

```bash
docker build -t egorggggg/cct-rag:latest .
```

Запуск:

```bash
docker run -d --name cct-rag -p 8010:8000 --env-file /path/to/cct-rag.env egorggggg/cct-rag:latest
```

Для public-версии можно запустить отдельный контейнер с другим entrypoint:

```bash
docker run -d --name cct-rag-public -p 8011:8000 --env-file /path/to/cct-rag.env egorggggg/cct-rag:latest uvicorn rag_web_public:app --host 0.0.0.0 --port 8000
```

Пример `/path/to/cct-rag.env`:

```text
OPENAI_API_KEY=your_real_key
OPENAI_BASE_URL=https://api.proxyapi.ru/openai/v1
OPENAI_MODEL=gpt-5.3-chat-latest
```

Перед сборкой Docker-образа убедиться, что рядом с кодом есть `npa/` и `УФАС/`, если образ должен содержать скрытый корпус внутри себя.

## Что считается текущим production-путём

Актуальный production-контур:

- `rag_web.py`
- `summary_rag.py`
- `simple_rag.py`
- `rag_support.py`
- `summary_prompts.py`
- `ufas_prompts.py`

Legacy-файлы вроде `main.py` и `app.py` не участвуют в текущем веб-пайплайне.
